Pre-Integration Discovery Checklist

Complete this checklist before writing a single line of code. Incomplete discovery is the #1 cause of integration delays. Every provider implements standards differently—don't assume standard behavior.

⚠️ Critical Warning
Skipping discovery items results in costly rework, missed go-live deadlines, and production incidents. All items must be resolved and documented before sandbox access is granted.

1️⃣ Discovery & Requirement Gathering

Core Services & Coverage
  • Supported Payment Services Confirm support for: payouts, collections, FX conversion, balance inquiry, virtual accounts, statement inquiry Required
  • Geographic & Currency Coverage Obtain complete list of supported countries, currencies, and transaction types per market Required
  • Volume & Settlement SLAs Document expected daily/monthly transaction volumes and settlement time-to-finality (T+0, T+1, T+3, etc.) Required
  • Provider Onboarding Timeline Confirm provider's onboarding duration, certification requirements, and support SLAs Required
  • Technical Documentation Collect API specs, OpenAPI/Swagger files, integration guides, test card numbers, and error code mappings Required
  • Rate Limit Configuration Clarify TPS limits, burst allowances, per-endpoint caps, and hard vs. soft limits Required

2️⃣ Authentication & Authorization

Auth Model & Credentials
  • Authentication Method Identify auth type: Client ID/Secret, OAuth2, mTLS, API Key, JWT, or custom. Get detailed flow documentation. Required
  • OAuth Flows & Token Expiry Confirm refresh token duration, authorization code grant requirements, and token rotation policies Required
  • Certificates & Key Exchange Obtain mTLS certificates, RSA keys, webhook signing secrets, and certificate expiry schedules Required
  • Callback & Redirect URIs Register all OAuth redirect URIs and webhook callback endpoints (including sandbox and production) Required
  • Token-Endpoint Rate Limits Clarify any separate rate limits on authentication endpoints to prevent auth storms Recommended
  • Encryption & Payload Signing Confirm if request/response encryption or signing is required; obtain algorithm specs and test samples Required

3️⃣ Connectivity & Network Setup

Infrastructure & Environment Configuration
  • Static IP Requirements Confirm if static IPs are needed for outbound requests; document IP ranges for both sandbox and production Required
  • VPN / VPC Peering Setup Request and validate VPN or VPC peering for sandbox and production; test connectivity before go-live Required
  • Ports & Protocols Document all required ports (443 for HTTPS, custom ports for SFTP, etc.) and protocol requirements Required
  • IP Allowlisting Determine if inbound/outbound IP allowlisting is required; document provider IP ranges for whitelisting Recommended
  • Sandbox Connectivity Validation Perform end-to-end connectivity test in sandbox before requesting production credentials Required
  • Test Credentials & Static IDs Obtain test API keys, merchant IDs, application IDs, OrgID, and all static identifiers Required

4️⃣ API & Webhook Behavior

Integration Patterns & Event Handling
  • Sync vs. Async API Behavior Clarify API response patterns: synchronous (instant response), asynchronous (eventual consistency), or hybrid Required
  • Webhook Documentation Obtain webhook event types, payload schemas, delivery guarantees, and signature algorithm details Required
  • Idempotency Strategy Align on idempotency key usage for outbound requests AND inbound webhook deduplication Required
  • Reconciliation Files Obtain reconciliation file formats (CSV, XML), delivery mechanism (SFTP, API), and frequency (daily, hourly) Required
  • Technical Kickoff Session Schedule and attend provider technical onboarding workshop with API team and DevOps Required

5️⃣ Testing & Certification

UAT & Production Readiness
  • Provider Test Scenarios Obtain happy-path, failure, and edge-case test scenarios; document all decline codes and error states Required
  • Sandbox Availability & Longevity Confirm sandbox is permanent or time-limited (some providers deactivate after 90 days); plan accordingly Required
  • Certification Checklist Confirm if provider requires sign-off on specific test cases or certification process before production Recommended
  • Production Lead Time Document required lead time from UAT sign-off to production enablement (typically 5-10 business days) Required

6️⃣ Operations & Support

Ongoing Maintenance & Escalation
  • Communication Channels Document primary, secondary, and emergency contact channels; confirm response SLAs Required
  • Maintenance Windows Clarify how maintenance notifications are communicated; document typical outage windows Required
  • Escalation Contacts Obtain technical, commercial, and executive escalation contacts with availability info Recommended
  • SLA Terms Document uptime SLAs, incident response times, and support hours (24x7, business hours, etc.) Required
✅ Discovery Completion Indicator
All connectivity items must be green before moving to sandbox. Incomplete network setup (VPN, IP allowlisting, certificate exchange) is the #1 cause of integration delays. Schedule a 30-minute kickoff call with your internal DevOps/Security team AND the provider's technical team to work through this together.

Payment Integration Architecture

A proven four-layer architecture for designing scalable, maintainable, and secure payment integrations.

Four-Layer Integration Stack

Layer 1: Presentation

Responsibility: User-facing payment collection

Components: Client SDKs, hosted payment pages, 3DS iframes, mobile payment sheets

Ownership: Frontend & Mobile teams

Layer 2: API Gateway

Responsibility: Request validation & routing

Components: Rate limiting, auth validation, idempotency enforcement, request transformation

Ownership: Platform/Infrastructure team

Layer 3: Processing Core

Responsibility: Transaction logic & business rules

Components: Fraud scoring (Sardine, Seon), rule engine, currency conversion, fee calculation

Ownership: Payment Engineering team

Layer 4: Settlement Layer

Responsibility: Banking rails integration

Components: Bank connectors, ledger, reconciliation, reporting, payout scheduling

Ownership: Payments Operations & Finance

Request Lifecycle

Every payment request follows this standardized flow:

  1. Client Initiation: Client submits payment via SDK or API with payment method details
  2. Gateway Validation: API Gateway validates JWT token, checks rate limits, enforces idempotency
  3. Fraud Scoring: Request routed to Processing Core for fraud detection and rule evaluation
  4. Network Routing: If fraud score passes threshold, transaction sent to appropriate bank connector
  5. Synchronous Response: Bank response mapped to canonical status and returned immediately to client
  6. Asynchronous Confirmation: Webhook fires for final settlement status (T+0 to T+3 depending on network)
💡 Best Practice: Canonical Status Mapping
Always map provider-specific status codes to an internal canonical enum. This abstraction prevents vendor lock-in, simplifies multi-provider support, and makes status transitions explicit and testable.

API Authentication & Security

Security is the foundation of every payment integration. This section covers authentication patterns, request signing, and mandatory controls.

OAuth 2.0 & JWT Authentication

Most modern payment APIs require a signed JWT in the Authorization header. Tokens expire after a short period (typically 15 minutes). Refresh tokens are long-lived (typically 30 days).

Control Requirement Priority
Token Storage Never in localStorage. Use secure server-side vaults (HashiCorp Vault, AWS Secrets Manager, Azure Key Vault) Required
Token Expiry Assume 15-minute access token TTL. Implement proactive refresh before expiry. Required
Client Secret Rotation Rotate every 90 days AND immediately upon suspected compromise Required
Scope Restriction Request minimum necessary scopes. Avoid wildcard permissions. Recommended
Token Logging Never log access/refresh tokens or secrets in application logs or monitoring systems Required
Separate Token Rate Limits Clarify if auth endpoint has separate rate limits to prevent auth storms Recommended

Request Signing (HMAC-SHA256)

High-value transactions (typically >$10,000) require an additional signature over the canonical request string to prevent tampering and replay attacks.

Canonical Request Format

METHOD\nPATH\nTIMESTAMP\nBODY_SHA256

Signature Computation

HMAC-SHA256(signing_secret, canonical_string)

Key Security Considerations

  • Timestamp Validation: Reject requests with timestamps older than 5 minutes to prevent replay attacks
  • Body Canonicalization: JSON bodies must be serialized consistently (sorted keys, no extra whitespace)
  • Secret Rotation: Rotate signing secrets on the same 90-day schedule as client secrets
  • Idempotency Integration: Include idempotency key in signed payload to prevent replay with different keys

Mandatory Security Controls

Control Implementation Standard
TLS Version TLS 1.2 or higher enforced on ALL endpoints PCI-DSS
Certificate Pinning Recommended for mobile SDKs to prevent MITM attacks Best Practice
PII Encryption at Rest All PII fields encrypted using AES-256 GDPR/CCPA
Audit Logs Retained for 7 years, immutable, with tamper detection SOX/PCI
Webhook Timeout Endpoints must respond within 5 seconds Operational
Webhook HTTPS Only All webhook endpoints MUST use HTTPS, including sandbox Security
🔒 Token Management Best Practice
Implement a token refresh loop that proactively refreshes tokens before expiry. Store tokens in a in-memory cache with automatic refresh on 401 errors. Never trust token TTL claims from clients—always validate server-side.

Core Integration Protocols

Universal API patterns for payment processing: payment intents, idempotency, and refunds.

Payment Intent Pattern

A PaymentIntent represents the complete lifecycle of a charge attempt. Always create server-side to prevent tampering.

PaymentIntent Status State Machine

Status Meaning Next States
requires_payment_method Awaiting payment method attachment → requires_confirmation
requires_confirmation Payment method attached, awaiting confirmation → processing, canceled
processing Authorization in-flight with network → succeeded, failed
succeeded Funds captured (terminal) —
failed Declined or errored (terminal) —
canceled Voided before capture (terminal) —
💡 State Machine Implementation
Implement explicit state machine logic for PaymentIntent statuses. Handle terminal states as irreversible. Never assume linear progression. Use webhooks for asynchronous transitions rather than polling.

Refunds & Reversals

Key Considerations

  • Partial Refunds: Supported by most providers. Specify the exact refund amount.
  • Refund Reasons: Use standardized codes: requested_by_customer, duplicate, fraudulent
  • Settlement Timing: Full refunds take 5-10 business days to appear on customer statements
  • Idempotency: ALWAYS use idempotency key to prevent accidental duplicate refunds
  • Metadata: Attach support ticket references for reconciliation tracking

Idempotency Key Strategy

Rule: Retain idempotency keys for at least 24 hours after request completes. Most providers cache responses for 24 hours.

Format: Use UUID v4 or merchant_id:request_id:attempt pattern for easy tracking

Scope: Must be unique per (merchant, endpoint, request_body) tuple to prevent unintended deduplication

Sandbox Stability for Continuous Integration

The sandbox environment is your foundation for long-term development. Unlike production, sandbox must be stable, predictable, and permanently available as you build and iterate continuously. This section covers strategies for maintaining sandbox integrity and scaling your integration sustainably.

🚨 Critical: Sandbox as Long-Term Foundation
Sandbox instability will compound integration pain as your system scales. Many teams discover sandbox issues only after weeks of integration work. Plan for sandbox to be your permanent testing environment, not a throwaway.

1️⃣ Sandbox Longevity & Availability

Confirming Sandbox Will Remain Available
  • Confirm Sandbox Permanence Critical: Ask provider explicitly: "Is sandbox available indefinitely, or only during integration phase?" Some providers deactivate sandbox 90 days after go-live. Required
  • Sandbox Data Purge Policy Document if/when sandbox data is purged (daily, weekly, monthly, never). Plan test data setup accordingly. Required
  • Maintenance Windows & Notifications Get schedule for sandbox maintenance. Understand how provider notifies of planned downtime. Document SLA expectations. Recommended
  • Sandbox Feature Parity with Production Confirm all production features are available in sandbox. Identify any limitations or differences. Document them. Required
  • Sandbox Credential Rotation Understand sandbox credential lifecycle. Are sandbox API keys permanent, or do they expire? Plan credential management accordingly. Recommended

2️⃣ Test Data Management for Continuous Integration

Stable test data is essential for reproducible testing across sprints and quarters. Build a test data strategy that survives purges and scales with your test suite.

Test Data Setup & Maintenance
  • Idempotent Test Data Setup Scripts Create scripts that can re-create test data (merchants, accounts, payment methods) from scratch. Run on every major test cycle. Commit to Git. Required
  • Test Merchant IDs & Virtual Accounts Create stable test merchant accounts that survive sandbox purges. Use clear naming: test-merchant-payments-team-001, test-merchant-qa-regression-001 Required
  • Test Payment Method Library Maintain a catalog of test cards/methods for all scenarios: success, insufficient funds, 3DS, network timeout, etc. Required
  • Sandbox FX Rates & Pricing Confirm if FX rates in sandbox match production or differ. If sandbox uses fixed rates, use them for testing stability. Document any differences. Recommended
  • Documentation of All Test Scenarios Keep a README in your test data repo listing all test cards, merchants, and what each tests. Update whenever you add new scenarios. Required

3️⃣ Sandbox Scenario Simulation & Determinism

Sandbox must behave deterministically. You need to be able to reproduce issues reliably without random variation or provider-side randomness interfering with your tests.

Deterministic Scenario Control
  • Provider Scenario Headers (if supported) Some providers allow headers like X-Sandbox-Scenario: decline_insufficient_funds to trigger specific outcomes. Document all available scenarios. Recommended
  • Settlement Delay Simulation If provider supports X-Sandbox-Settlement-Delay: 3d, use it to test async settlement flows without waiting. Recommended
  • Webhook Delivery Testing Test webhook delivery in sandbox: success, delayed delivery, duplicate delivery. Some providers have webhook simulation tools. Recommended
  • 3DS Challenge Simulation Use sandbox 3DS test cards to reliably trigger 3DS challenges without human intervention. Test both success and failure paths. Required
  • Network Latency/Timeout Simulation If provider supports, simulate timeouts and degraded network. Test your retry logic at scale without affecting production. Recommended

4️⃣ Continuous Integration & Automated Sandbox Testing

As your system evolves, sandbox testing must be automated in your CI/CD pipeline. This catches integration regressions early and ensures sandbox stability is validated on every merge.

CI/CD Sandbox Integration
  • Automated Sandbox Smoke Tests Run minimal happy-path tests on every PR: create payment intent, confirm transaction, verify webhook. Should complete in <1 minute. Required
  • Regression Test Suite Build comprehensive test suite covering all API operations. Run nightly or on-demand. Document test coverage per endpoint. Required
  • Idempotency Key Validation Automated test: send same request twice with same idempotency key, verify identical response returned. Required
  • Webhook Delivery Verification Automated webhook listener: trigger transaction, verify webhook arrives within expected time window. Required
  • Token Refresh Testing Automated test: verify token refresh works, expired tokens are rejected, new tokens work. Required
  • Rate Limit Handling Validation Automated load test: hit sandbox at 110% expected TPS, verify rate limiting doesn't drop transactions. Recommended

5️⃣ Monitoring & Health Checks for Sandbox

Sandbox uptime issues should be detected immediately, not discovered when your team starts testing. Monitor sandbox like you monitor production.

Sandbox Observability
  • Synthetic Health Check Every 5 Minutes Ping sandbox auth endpoint every 5 minutes. Alert if health check fails for 2+ consecutive checks. Logs to monitoring system. Required
  • API Latency Tracking Log p50, p95, p99 latencies for each endpoint. Alert if p99 latency exceeds baseline by >50%. Recommended
  • Webhook Delivery Monitoring Track webhook delivery success rate, average delivery time. Alert if delivery time exceeds 10 seconds. Recommended
  • Sandbox Availability Dashboard Public dashboard (internal) showing sandbox uptime, recent incidents, maintenance windows. Share with team. Recommended
  • Incident Log for Sandbox Issues Document any sandbox outages, data corruption, or provider bugs. Share post-mortems with provider. Recommended

6️⃣ Scaling: From Single Team to Enterprise

As multiple teams use sandbox (payments, mobile, web, operations), sandbox must support concurrent testing without interference.

Multi-Team Sandbox Usage
  • Isolated Test Merchant Accounts Create separate test merchants per team: test-mobile-team, test-web-team, test-api-team. Prevent cross-contamination. Required
  • TPS Quota Per Team If sandbox has fixed TPS limits, allocate quota: 5 TPS per team for smoke tests, central load test pool with 10 TPS. Recommended
  • Sandbox Reservation System For load tests (110% TPS), require booking 48 hours in advance. Prevents chaos from concurrent load tests. Recommended
  • Shared Runbook & FAQ Document sandbox quirks, common issues, and solutions. Update as teams encounter issues. Required

7️⃣ Sandbox-Specific Testing Strategy

Sandbox tests differ from production tests. Your test strategy must account for sandbox's eventual-consistency and async nature.

Test Categories

Test Type Frequency Responsibility Notes

Regression Suite Nightly + before release QA team All API operations, decline codes, webhooks. ~30 min runtime.

💡 Best Practice: Sandbox as Truth
Treat sandbox as your "source of truth" for integration behavior. If behavior differs between sandbox and production, investigate immediately. This often uncovers configuration or environment setup issues before they hit production.

Error Handling & Retry Logic

Robust error handling separates production-grade integrations from fragile ones. Follow RFC 7807 Problem Details format. Never expose stack traces in production.

HTTP Status Code Reference & Retry Strategy

Code Meaning Action Retry?
200 Success Process response ✗
400 Bad request / validation error Fix payload, don't retry ✗
401 Unauthenticated (token expired) Refresh token, retry once ✓ (once)
402 Card declined Retry only for soft declines ✓ (conditional)
403 Forbidden (insufficient scope) Don't retry, escalate ✗
409 Idempotency key conflict Reuse existing response ✗
422 Unprocessable entity Fix request, don't retry ✗
429 Rate limited Read Retry-After, wait, re-enqueue ✓
500 Internal server error Exponential back-off with jitter ✓
503 Service unavailable Exponential back-off, circuit breaker ✓

Exponential Back-off Formula

delay = base_delay × (2^attempt) + jitter

where:
  base_delay = 1 second
  max_attempts = 5
  jitter = random(0, delay × 0.25)
⚠️ Never Retry These Errors
Do NOT retry on HTTP 400, 403, 409, or 422. These are client-side errors that won't resolve with repetition. Retrying wastes resources and may trigger rate limits.

FX Rate Locking & Currency Management

FX rate fluctuations can cause disputes, reconciliation headaches, and regulatory issues if not managed carefully. This section covers rate locking strategies, validity windows, and handling currency conversions at scale.

🚨 Critical: Rate Locking Prevents Customer Disputes
A customer expects to pay exactly $100 USD, not $100.50 due to FX slippage between quote and payment. Without explicit rate locking, you'll face chargebacks and disputes. Document your rate policy clearly.

1️⃣ FX Rate Locking Strategy

FX rates must be locked at a clear, documented point in the transaction lifecycle. Ambiguity here causes customer disputes and reconciliation failures.

Rate Locking Decision Points
  • Define Rate Locking Point Explicitly Critical decision: Rate locked at quote time, payment intent creation, or confirmation? Document in customer-facing UI. Required
  • Rate Validity Window How long is a quoted rate valid? Typical: 30s for instant payments, 5 minutes for checkout, 1 hour for quote sheets. Document in API. Required
  • Rate Spread / Margin Do you apply a markup (e.g., mid-market rate + 0.5%)? Document spread in terms of service and UI. Required
  • Rate Refresh Strategy If rate quote expires, do you re-quote or fail the transaction? Define fallback behavior. Required
  • Provider Rate Refresh Frequency Confirm provider FX rate sheet refresh schedule (typically every 10-60 seconds). Align with your rate locking window. Required

2️⃣ FX Rate Fluctuation Management

FX rates move constantly. You must decide: absorb the loss, pass to customer, or reject transactions outside a tolerance band?

Fluctuation Handling Strategies
  • Tolerance Band Definition Define acceptable rate fluctuation: ±0.1%, ±0.5%, ±1%? Use mid-market as baseline. Communicate to customers. Required
  • Out-of-Band Fluctuation Handling Decision: If rate moves >tolerance: (A) auto-reject & ask for re-quote, (B) absorb delta, (C) pass to customer? Choose one, document it. Required
  • Slippage Logging & Monitoring Log every transaction: quoted rate, locked rate, actual settlement rate. Track slippage distribution. Alert if >1% slippage occurs. Required
  • Volatility Hedging (Enterprise) For high-volume corridors, consider FX forward contracts to hedge volatility exposure. Coordinate with finance team. Optional
  • Weekend / Holiday Rate Gaps FX markets close on weekends. Use Friday EOD rates for weekend transactions, or disable cross-border payments during gaps. Recommended

3️⃣ Rate Locking Implementation Patterns

Pattern 1: Quote-Time Locking (Recommended for B2C)

Flow: Customer enters amount → System fetches live rate → Lock rate for 30 seconds → Customer confirms → Payment uses locked rate

Pros: Transparent to customer, prevents surprises, simple implementation

Cons: Must handle quote expiry, requires fast checkout

Pattern 2: Payment Intent Locking (Recommended for B2B)

Flow: Create PaymentIntent → Rate locked at this moment → Customer can pay within 1 hour at locked rate → Settlement uses locked rate

Pros: Longer validity window, reduces customer re-quote spam, clearer audit trail

Cons: Rate locked before payment confirmed (market risk)

Pattern 3: Confirmation-Time Locking (Riskiest)

Flow: Payment confirmed → Rate fetched at confirmation moment → Immediate settlement

Pros: Always latest rate

Cons: Customer doesn't know final amount until confirmation—causes friction & disputes. Avoid unless you have very low-latency execution.

4️⃣ Stablecoin Integration for FX Mitigation

Stablecoins (USDC, USDT, stablecoin-backed currencies) can reduce FX volatility for cross-border payments. Consider them as settlement currency for select corridors.

Stablecoin Considerations
  • Stablecoin Selection Decision: Which stablecoins to support? (USDC, USDT, EUROC, SGDC?) Evaluate liquidity, regulatory status, issuer trustworthiness. Required
  • Stablecoin Corridors & Use Cases Where do stablecoins make sense? Cross-border B2B transfers (low latency, instant settlement). Consumer P2P (emerging markets with high inflation). Recommended
  • Stablecoin ↔ Fiat Conversion Rates Stablecoins aren't perfectly pegged. Monitor USDC/USD spread on exchanges. Lock spread when quoting conversion from/to fiat. Recommended
  • Blockchain Settlement Risk Stablecoins settle on-chain (Ethereum, Polygon, Solana, etc.). Account for network congestion, variable gas fees, finality delays. Required
  • Regulatory Compliance Stablecoin regulations vary by jurisdiction (EU MiCA, Singapore MAS, US state licenses). Consult compliance before launch. Required
  • Custody & Private Keys Who holds private keys for stablecoin wallets? (Your company, custodian, user's self-custody?) This affects security, compliance, and liability. Required
  • Fallback to Fiat Rails Stablecoins fail (network outage, issuer event). Always have fiat settlement fallback. Design dual-settlement architecture. Required

5️⃣ FX Rate Data & Real-Time Updates

Rate Data Strategy
  • Rate Source Redundancy Never rely on single provider for rates. Use primary + fallback sources (Bloomberg, Xe.com, provider API, internal model). Required
  • Rate Update Frequency How often do you refresh rates? Every 10s, 30s, 60s? Faster = better customer experience but higher data costs. Required
  • Rate Staleness Detection Alert if rate data becomes stale (older than 2x update frequency). Disable payments rather than quote stale rates. Required
  • Rate Caching & TTL Cache rates in-memory with explicit TTL (e.g., 30 seconds). Expire cache aggressively to prevent stale quotes. Recommended
  • Rate Outlier Detection Detect suspicious rate movements (e.g., 10% jump from previous rate). Alert ops. Consider rate as potentially erroneous. Recommended

💡 FX Rate Best Practice: The "Quote → Lock → Settle" Pattern
Always store three distinct rates: (1) quoted_rate shown to customer, (2) locked_rate committed to customer, (3) settlement_rate applied by provider. This triple audit trail resolves 99% of FX disputes and makes reconciliation bulletproof.

Stablecoin & Alternative Currency Strategies

Beyond traditional fiat currencies, stablecoins and emerging currencies (CBDCs, regional digital currencies) offer settlement alternatives with different tradeoffs. This section covers architecture patterns, regulatory considerations, and when to use them.

1️⃣ Stablecoin vs. Fiat Settlement Comparison

Dimension Traditional Fiat (USD, EUR, SGD) Stablecoins (USDC, EUROC) CBDCs (e-SGD, e-EUR)
Settlement Speed 1-3 business days (SWIFT) or T+1 (ACH) 15-60 seconds on-chain Instant (programmatic)
Cost $0.50-$25 per payment (bank fees) $0.01-$5 (variable gas fees) $0 (government-subsidized)
Regulatory Maturity Fully established globally Emerging (MiCA in EU, evolving in US/APAC) Pilot phase (limited jurisdictions)
Counterparty Risk Bank/central bank risk Issuer risk (Circle, Tether, PayPal) Central bank (very low)
Liquidity Unlimited (interbank markets) Good for major pairs (USDC), emerging for regional Limited (new market)
Network Dependency None (traditional banking network) Blockchain network (Ethereum, Polygon, etc.) Central ledger (government-run)
Volatility Risk FX volatility (between fiat pairs) Stablecoin peg risk + underlying FX volatility None (government-backed peg)
Use Case Fit All use cases (incumbent standard) Cross-border B2B, emerging markets, crypto-native Domestic payments, government interop

2️⃣ Multi-Currency Architecture for Stablecoins

If you support both fiat and stablecoins, your architecture must handle conversion, routing, and fallback seamlessly.

Stablecoin Integration Architecture
  • Dual Settlement Layer Layer 4 (Settlement) must support multiple rails: fiat (SWIFT/ACH), stablecoins (blockchain), CBDCs (when available). Route based on user preference. Required
  • Currency Conversion Engine Fiat ↔ Stablecoin conversion needs its own logic: check stablecoin peg, apply spread, handle slippage separately from FX. Required
  • Atomic Swap / Bridge Architecture User sends USD, system converts to USDC, settles on-chain, converts back to SGD at destination. Coordinate with liquidity partners. Recommended
  • Failover from Stablecoin to Fiat If blockchain network is congested or stablecoin peg breaks, automatically failover to fiat settlement. Must be seamless to user. Required
  • Wallet Infrastructure for Stablecoins Do users have wallets? (Self-custody, managed wallets, sub-accounts?) Define custody model before launch. Required

3️⃣ Stablecoin Liquidity & Corridor Selection

Stablecoins work best for specific corridors where liquidity is deep. Don't launch support for all currency pairs simultaneously.

Corridor Launch Strategy
  • Liquidity Analysis by Corridor Research stablecoin liquidity for target corridors: USDC/USDT/USDC.e on Ethereum, Polygon, Solana? Check 24h volume. Required
  • Blockchain Network Selection Which blockchain(s) for stablecoins? Ethereum (high fees, high liquidity), Polygon (lower fees, growing), Solana (fast, but less mature)? Required
  • On-Ramp / Off-Ramp Partners Need liquidity partners to convert fiat ↔ stablecoin at scale: Kraken, Fireblocks, Airtm, MoonPay, etc. Evaluate rates & speed. Required
  • Pilot Corridors Start with 1-2 high-liquidity corridors (e.g., USDC SGD→PHP or USD→USDC stablecoin in emerging markets). Expand after proving model. Recommended
  • Spread vs. Liquidity Tradeoff Stablecoin routes have lower spread but higher operational complexity. Quantify: is 0.1% spread saving worth the 10% higher engineering cost? Recommended

4️⃣ Blockchain & Crypto-Specific Risks

Stablecoins live on blockchains. You inherit blockchain risks: network congestion, failed transactions, network forks, smart contract bugs.

Blockchain Risk Mitigation
  • Gas Fee Volatility On Ethereum, gas fees can spike 10x during network congestion. Build dynamically adjustable fee calculation. Set cap: refuse transaction if gas > X% of payment. Required
  • Transaction Finality Monitoring Track transaction status: pending → confirmed (1 block) → final (12+ blocks). Don't assume finality until >12 blocks on Ethereum. Required
  • Failed/Reverted Transaction Handling Transactions can fail after gas is spent (contract revert, out-of-liquidity). Detect & retry with higher gas, or escalate. Required
  • Network Fork / Reorganization Risk Blockchains can reorganize (reorg). After 12 blocks, risk is negligible, but policy: how many blocks = settled? Recommended
  • Smart Contract Audit Any custom contracts? Get external audit from reputable firm (OpenZeppelin, Trail of Bits, Certik). Required
  • Bridge Risk (If Using Cross-Chain) Bridges (Polygon to Ethereum, etc.) have ~1-10% annual loss risk. Mitigate by using reputable bridges (Aave Portal, Stargate). Recommended

5️⃣ Regulatory & Compliance for Stablecoins

Jurisdiction Regulatory Status Key Requirements
EU (MiCA) Regulated (MiCA effective 2024) Crypto service provider license required. USDC/stablecoins must be E-Money Tokens or Asset Referenced Tokens. Full KYC/AML.
Singapore Regulated (MAS Payment Services Act) Money Services License required for stablecoin issuance or provision. USDC generally allowed. KYC/AML mandatory.
US Fragmented (federal + state) State Money Transmitter License (varies per state). Federal reporting for >$10k. SEC monitoring for issuers.
Hong Kong Emerging (SFC oversight) Stablecoin trading platforms require SFC license. USDC trading allowed. Clear KYC/AML policies.

⚠️ Regulatory Risk: Stablecoin regulations change rapidly
What's allowed today may be prohibited tomorrow. Have legal counsel monitor regulatory developments. Build architecture to quickly disable stablecoin routes if regulations shift. Never commit >3-6 months ahead on stablecoin roadmap without legal sign-off.

Common Integration Challenges & Mitigation

Bank integrations are rarely frictionless. This section documents real-world challenges your team will face and proven strategies to overcome them.

1️⃣ Authentication & Credential Issues

Challenge: OAuth Token Expiry & Refresh Token Complications
  • Problem: Refresh token expires without notice Symptom: Integration works for days, then sudden 401 Unauthorized on all requests. Root cause: Refresh token silently expired (many providers have 30-90 day TTL). Mitigation: Track token and refresh token expiry separately. Implement alerting when refresh token has <7 days until expiry. Have process to re-authenticate before expiry. Required
  • Problem: Provider rotates API keys without notification Symptom: Valid requests suddenly fail with 401. Root cause: Provider rotated client secret (typically every 90 days), and you didn't know. Mitigation: Ask provider for explicit rotation schedule. Rotate YOUR client secret proactively (don't wait for provider). Have fallback to secondary credentials during rotation window. Build monitoring to detect auth failures <1 minute after secret rotation. Required
  • Problem: Sandbox vs. Production credentials get confused Symptom: Tests pass in sandbox, but production environment uses old/wrong API key. Root cause: Credentials stored in wrong place or not rotated together. Mitigation: Use environment-specific credential vaults (AWS Secrets Manager per env). NEVER check API keys into Git. Implement credential rotation script that updates ALL environments atomically. Test credential validity in health check before processing first transaction. Required

2️⃣ API & Protocol Mismatch Issues

Challenge: API Documentation Mismatches Implementation
  • Problem: API spec says field is required, but provider doesn't actually enforce it Symptom: You include customer_id (supposedly required), but provider ignores it silently. Later, reconciliation shows transactions with null customer_id. Mitigation: Test every required field with it missing. Document observed behavior vs. spec. Ask provider to clarify. Build defensive code: make fields optional in parser, but strict in business logic. Log warnings when required fields are missing. Recommended
  • Problem: API returns different response schema than documented Symptom: You parse transaction.status as string, but it's sometimes an object with code and reason fields. Mitigation: Use explicit JSON schema validation (json-schema library). Log and alert on schema mismatches. Never assume response structure—always validate before parsing. Keep detailed logs of all response variations you encounter. Required
  • Problem: Undocumented rate limits cause cascading failures Symptom: Provider says 500 req/min limit, but actual limit is 100 req/min per merchant. You hit limit on one merchant and cascade to others. Mitigation: Load test extensively in sandbox before production. Use exponential backoff with jitter from day 1 (not as afterthought). Implement per-merchant and global rate limit tracking. Alert when approaching 80% of limit. Required

3️⃣ Webhook Delivery & Timing Issues

Challenge: Webhooks Arrive Out-of-Order or Never Arrive
  • Problem: Webhook for settlement arrives BEFORE synchronous response confirms payment Symptom: Your system calls POST /confirm-payment, gets a timeout/503, then 5 seconds later receives webhook saying payment succeeded. Did it succeed or fail? Mitigation: Never rely solely on sync response or webhook—require BOTH for settlement. Treat timeout as "unknown" state. Implement idempotency strictly: if you receive webhook for payment you didn't initiate, investigate. Build reconciliation to compare API state vs. webhook state daily. Required
  • Problem: Webhooks arrive out of order Symptom: You receive payment_intent.succeeded before payment_intent.processing. Your state machine breaks. Mitigation: NEVER assume webhook order. Store event timestamps. Process by timestamp, not by arrival order. Build state machine to handle events arriving in any order. Log out-of-order events. Alert if order is severely jumbled (may indicate provider bug). Required
  • Problem: Webhooks never arrive (total black hole) Symptom: Payment settles on provider side, but webhook never fires. Your system waits forever. Mitigation: Implement timeout: if webhook hasn't arrived within SLA window (e.g., 5 minutes for instant payments), query provider API to check status. Reconciliation cron job: every hour, compare received webhooks vs. actual transactions in provider API. Log discrepancies. Required

4️⃣ FX & Multi-Currency Challenges

Challenge: FX Rate Movements Cause Unexpected Losses or Disputes
  • Problem: Rate expires but you don't detect it Symptom: You quote customer USD/SGD rate valid for 60 seconds. Customer takes 90 seconds to confirm. You submit payment at old rate, but provider applies current rate (worse for you). Mitigation: Always check rate validity window BEFORE submitting payment. If expired, reject and ask for new quote. Log rate expiries. Alert if customer exceeds typical checkout time (may indicate UX friction). Required
  • Problem: Stablecoin peg breaks unexpectedly Symptom: USDC drops to $0.95, causing your stablecoin conversion math to break. You quoted at 1:1, but get $95 instead of $100. Mitigation: Monitor stablecoin spot price on exchanges. If peg deviates >0.5%, disable stablecoin routes temporarily. Implement fallback to fiat immediately. Alert ops. Required
  • Problem: Weekend / holiday FX market closures Symptom: You send cross-border payment Friday 6 PM, rate freezes, settlement doesn't happen until Monday, rate has moved 1%. Mitigation: Disable cross-border payments 2 hours before FX market close on Friday. Use Friday EOD rate for weekend transactions (lock it). Communicate to customers: "Cross-border payments submitted after X will settle Monday at Friday's rate." Recommended

5️⃣ Sandbox & Production Environment Discrepancies

Challenge: Code Works in Sandbox But Fails in Production
  • Problem: Sandbox has different error messages than production Symptom: Sandbox returns "error": "card_declined", but production returns "decline_code": "insufficient_funds". Your error handler breaks. Mitigation: Normalize all provider error responses to canonical format immediately after receiving. Don't parse provider errors directly—map them through a translator layer. Test error handling with actual production decline codes (ask provider for test scenarios). Required
  • Problem: Production rate limits are much stricter than sandbox Symptom: Sandbox caps at 50 TPS but production is 200 TPS per merchant (not global). You hit limit on merchant 1 thinking it's global. Mitigation: Ask provider for explicit rate limit structure and per-merchant vs. global caps. Load test production tier early (at least with staging credentials). Monitor 429 errors closely in production first week. Required
  • Problem: Webhook delivery behavior differs between sandbox and production Symptom: Sandbox delivers webhooks within 1 second. Production takes 30+ seconds. You implement timeout of 10 seconds based on sandbox behavior. Mitigation: Set conservative timeouts (5-10 min) for webhook processing, not based on sandbox. Test webhook delivery latency in production staging for a week before main launch. Document observed SLAs. Recommended

6️⃣ Operational & Support Challenges

Challenge: Provider Support Is Slow, Unresponsive, or Unhelpful
  • Problem: Critical issue (payment failures) takes 24+ hours to resolve via support Mitigation: NEVER rely on email support for critical issues. Require escalation phone line in contract. Have named technical escalation contacts (not just generic support queue). Schedule regular syncs (weekly or bi-weekly) with provider's technical team once in production. Keep relationship warm—they'll prioritize you. Required
  • Problem: Provider makes breaking API changes without warning Mitigation: Monitor provider's API changelog/status page religiously. Join their Slack/Discord community to hear about changes early. Version your API calls explicitly (use /v2/ not /). Subscribe to all deprecation notices. Have 6-month notice requirement in contract for breaking changes. Required
  • Problem: You don't understand provider's error, and they're not helpful in debugging Mitigation: Build comprehensive logging from day 1. Log EVERY request (sanitized) and response. When you hit error, you have full context to debug locally. Don't wait for provider support. Share logs with provider when escalating—makes their job easier and speeds resolution. Required

7️⃣ Reconciliation & Settlement Challenges

Challenge: API Says Payment Succeeded, But Settlement Never Arrives
  • Problem: Transaction marked "succeeded" but funds never appear in bank account Symptom: API says payment_intent.succeeded, customer sees charge, but bank doesn't show deposit for days (or ever). Mitigation: Reconciliation is NON-NEGOTIABLE. Run daily: compare all "succeeded" API transactions vs. bank settlement file. Flag missing transactions immediately. Escalate >$1000 discrepancies within 1 hour. Have process to refund + investigate missing settlements. Required
  • Problem: Settlement amount differs from API amount Symptom: You initiated payment for $1000, received webhook saying succeeded for $1000, but bank deposits $985. $15 missing (provider's fee?). Mitigation: Clearly document all fee structures upfront. Ask provider: where are fees deducted? (From amount or charged separately?) Log fee information in payment metadata. Reconciliation should account for fees. If fee differs from expected, investigate immediately. Required

8️⃣ Scaling & Performance Challenges

Challenge: Integration Breaks When Load Increases
  • Problem: Sudden traffic spike triggers provider's rate limiting Symptom: You launch marketing campaign, get 10x traffic, payment API returns 429 (rate limited). Mitigation: Negotiate higher TPS tiers BEFORE launching growth initiatives. Implement client-side throttling (token bucket) to stay under limit. Queue requests when approaching limit. Alert ops when hitting 80% of limit. Required
  • Problem: Webhook delivery becomes unreliable under load Symptom: Under normal load, webhooks arrive 99.9% of time. During peak traffic, 5% are lost. Mitigation: Implement webhook queue (not inline processing). Store incoming webhooks to database before processing. Process async from queue. This prevents timeouts and ensures all webhooks are logged. Required
  • Problem: Provider's infrastructure can't handle your volume Symptom: You reach 50 TPS (well under your 200 TPS tier), but provider's API becomes very slow or times out. Mitigation: Stress test provider in sandbox at >100% expected peak. Document observed degradation curve. If provider degrades at your expected volume, escalate immediately. May need to use different provider or negotiate special handling. Required
💡 Universal Challenge Mitigation Strategy
For EVERY integration challenge: (1) Implement comprehensive logging, (2) Build monitoring/alerting early, (3) Design degradation paths (when feature fails, what's the fallback?), (4) Automate recovery where possible, (5) Document incident response procedures. Prevention + quick recovery beats trying to prevent all issues.

Payment integrations operate in a heavily regulated environment. Engage qualified legal and compliance professionals before launching in new markets.

PSD2 / SCA EU/EEA payments Strong Customer Authentication (2FA) required. Exemptions for low-value (<€30) and trusted beneficiaries.

Production Readiness Checklist

Before requesting production access, verify every item below is complete. This checklist should be signed off by Engineering Lead, Security Team, and Compliance Team.

🔐 Authentication & Security

  • OAuth flow tested in sandbox with token refresh Verify token expiry handling, refresh token rotation, and re-authentication flows
  • Request signing validated for high-value transactions Test HMAC-SHA256 signing, timestamp validation, and signature verification
  • Client secret rotation procedure documented Document 90-day rotation schedule and emergency rotation process
  • IP allowlisting configured for production All static IPs registered with provider, VPN/VPC peering validated
  • Webhook signature verification implemented Verify HMAC-SHA256 signatures, reject unverified payloads, test with provider test events

🎯 Functionality & API Coverage

  • Happy path tested for all supported payment methods Cards, bank transfers, digital wallets—test successful completion for each
  • All decline codes handled appropriately Test insufficient funds, expired card, 3DS required, etc. Verify user messaging
  • Refund flow tested (full and partial) Test refund initiation, status tracking, and settlement timing
  • 3D Secure flow tested end-to-end Test 3DS challenge, OTP verification, and redirect flows
  • Multi-currency support validated (if applicable) Test currency conversion, FX rate locking, and settlement in local currency

⚡ Reliability & Error Handling

  • Exponential back-off retry logic implemented Test with simulated delays and failures; verify jitter prevents thundering herd
  • Idempotency keys used for all mutating requests Test duplicate request handling; verify same response returned within cache window
  • Webhook delivery retry & deduplication logic Store event IDs, discard duplicates, respond within 5 seconds, process async
  • Circuit breaker configured for provider outages Fail fast when provider is down; queue requests for retry; monitor health
  • Error handling tested for all RFC 7807 error codes Verify user-friendly error messages, no stack traces, proper logging

📊 Performance & Scalability

  • TPS requirements calculated and tier negotiated Use formula: Peak TPS = (Daily Transactions / Active Hours) × 3.0; document with provider
  • Load test passed at 110% of expected peak TPS Run sustained load tests in sandbox; verify no dropped transactions
  • p99 latency < 300ms at peak load Monitor and log latency percentiles; alert if approaching limits
  • Rate limit handling tested and monitored Test HTTP 429 responses; implement client-side throttling (token bucket / leaky bucket)

🛡️ Compliance & Security

  • PCI-DSS scope assessment completed Confirm no card data touches your systems; use tokenization or hosted payment pages
  • GDPR/CCPA data handling review completed Explicit consent for PII; implement right-to-erasure; verify data residency controls
  • Audit logging enabled for all payment operations Log all transactions, refunds, webhooks; retain for 7 years; immutable storage
  • Security review completed by internal security team Sign-off from security team on auth, encryption, and data handling

🚀 Operations & Monitoring

  • Monitoring and alerting configured for all endpoints Alert on error rate spike, latency degradation, 429 responses, webhook delays
  • Runbooks documented for common failure scenarios Token refresh failures, webhook delivery failures, provider outage procedures
  • Escalation contacts documented and verified Engineering, security, operations, and provider technical/commercial contacts
  • Incident response plan tested Run tabletop exercise for payment system outage; verify communication flow
  • Reconciliation process validated Daily reconciliation of API transactions vs. settlement; document discrepancies
